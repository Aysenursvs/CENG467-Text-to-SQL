\section{Introduction}
\label{sec:intro}

%% AÇIKLAMA: Giriş şu sırayı takip etmeli:
%%   Paragraf 1: Genel motivasyon — neden NL-to-SQL önemli?
%%   Paragraf 2: Mevcut zorluklar — neden bu problem hala açık?

Natural language interfaces to databases (NLIDB) allow users to query structured
data using plain English, eliminating the need for SQL expertise~\cite{[NLIDB_REFERENCE]}.
Text-to-SQL parsing, which maps natural language questions to executable SQL queries,
is a central challenge in this domain and has seen significant progress with the
advent of large pre-trained language models~\cite{[SURVEY_REFERENCE]}.

%% AÇIKLAMA: Mevcut zorlukları tartış. Aşağıdaki cümleler taslaktır, kendi cümlelerinle yaz.
Despite recent advances, Text-to-SQL remains challenging due to (1) the compositional
nature of SQL, (2) the need to correctly link question entities to database schema
elements (schema linking), and (3) the difficulty of generalizing across unseen
database schemas~\cite{Yu2018Spider}.
Cross-domain benchmarks such as Spider~\cite{Yu2018Spider} have highlighted
these challenges by evaluating models on databases not seen during training.

%% AÇIKLAMA: Bu çalışmanın katkısını listele. "We" ile başla.
In this paper, we make the following contributions:
\begin{itemize}
  \item We fine-tune [MODEL ADI] with instruction tuning on the Spider benchmark,
        incorporating explicit schema encoding to help the model understand
        database structure.
  \item We systematically compare three schema serialization formats and analyze
        their impact on execution accuracy via ablation study.
  \item We evaluate our approach against two baselines — zero-shot prompting and
        [BASELINE 2 ADI] — using execution accuracy and exact match metrics.
  \item We provide a detailed error analysis of failure cases, categorized by
        error type, along with a discussion of ethical considerations.
\end{itemize}

\subsection{Problem Formulation}
%% AÇIKLAMA: Problemi matematiksel/formal olarak tanımla.
%% Giriş ve çıkış nedir? x (soru), S (şema), y (SQL) notasyonunu kullan.

Given a natural language question $q$ and a relational database schema
$\mathcal{S} = \{T_1, T_2, \ldots, T_n\}$ where each table $T_i$ contains
a set of columns $C_i = \{c_{i,1}, \ldots, c_{i,m}\}$, the goal of Text-to-SQL
parsing is to generate an executable SQL query $y$ such that executing $y$
on the target database $\mathcal{D}$ returns the correct answer to $q$:

\begin{equation}
  y^* = \arg\max_{y} P(y \mid q, \mathcal{S}; \theta)
  \label{eq:objective}
\end{equation}

\noindent where $\theta$ denotes the model parameters.

\subsection{Schema Serialization Strategies}
%% AÇIKLAMA: Şemanın modele nasıl serileştirildiğini anlat.
%% 3 farklı format dene ve burada tanımla (ablation için de gerekli).

A key challenge in Text-to-SQL is representing the database schema as text
for input to a language model.
We investigate three serialization formats:

\noindent\textbf{Format A — Plain text listing:}
\begin{lstlisting}[language=SQL, caption={Format A: Plain text schema representation}]
Table: students | Columns: id, name, age, gpa
Table: courses  | Columns: id, title, credits
FK: enrollments.student_id -> students.id
\end{lstlisting}

\noindent\textbf{Format B — CREATE TABLE syntax:}
\begin{lstlisting}[language=SQL, caption={Format B: CREATE TABLE schema representation}]
CREATE TABLE students (
  id INT PRIMARY KEY,
  name TEXT, age INT, gpa FLOAT
);
\end{lstlisting}

\noindent\textbf{Format C — Compact notation:}
\begin{lstlisting}[language=SQL, caption={Format C: Compact schema representation}]
students(id[PK], name, age, gpa) |
courses(id[PK], title, credits)
\end{lstlisting}

%% AÇIKLAMA: Bu formatları neden seçtiğini bir cümleyle açıkla.
Format B mimics the input seen during code-oriented pre-training and is expected
to align best with code-focused models, while Format A prioritizes readability
and Format C minimizes input token count.

\subsection{Ethical Considerations}
%% AÇIKLAMA: Bu alt bölüm rubrikte 5 puan. Mutlaka doldurulmalı.
%% Hallucination riski, SQL injection, bias konularını tartış.

Deploying Text-to-SQL systems in production introduces several risks.

\noindent\textbf{Hallucination:} The model may generate syntactically valid
SQL that queries non-existent columns or tables, silently returning incorrect
results. Users without SQL expertise cannot detect such errors.

\noindent\textbf{SQL Injection Risk:} While the model operates on trusted schema
inputs in our setting, integrating user-provided free-text into SQL generation
pipelines could introduce injection vulnerabilities if outputs are not sanitized.

\noindent\textbf{Domain Bias:} Spider's training data skews toward English and
standard business domains (universities, music, sports). Models trained on Spider
may perform poorly on multilingual queries or specialized domains such as
healthcare or legal databases, raising fairness concerns for non-English-speaking
or domain-specific users.
