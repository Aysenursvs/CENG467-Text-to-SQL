\section{Project Title, Group Members, and Problem Definition}

%%>> Bu bölümde projeyi kısaca tanıtın. Aşağıdaki paragraf taslaktır,
%%>> kendi cümlelerinizle genişletin. Hedefiniz 150-200 kelime.

This project addresses the task of \textbf{Text-to-SQL semantic parsing},
which involves automatically translating natural language questions into
executable SQL queries over relational databases.

The task is important because it enables non-expert users to query structured
databases without knowledge of SQL syntax, a capability with broad practical
applications in business intelligence, healthcare, and data analysis.

We work on the \textbf{Spider}~\cite{Yu2018Spider} benchmark, a large-scale
cross-domain dataset containing [8,659] training and [1,034] development
question–SQL pairs across 200 databases.
The input to the system is a natural language question paired with a database
schema; the output is a syntactically valid and semantically correct SQL query.

The task is technically challenging for several reasons.
First, the model must correctly identify which tables and columns are relevant
to the question (\textit{schema linking}).
Second, it must generalize to database schemas unseen during training,
requiring compositional reasoning rather than pattern memorization.
Third, SQL supports complex constructs such as nested subqueries, aggregations,
and multi-table joins that are difficult to generate reliably.

\noindent\textbf{Data Source.}
The Spider dataset is publicly available and was downloaded from the official
Yale Semantic Parsing repository.
The dataset includes \texttt{train\_spider.json}, \texttt{dev.json},
\texttt{tables.json} (schema definitions), and a \texttt{database/} directory
containing SQLite files for execution-based evaluation.

\noindent\textbf{Input/Output Format.}
Each data instance contains a natural language question, a database identifier,
and a gold SQL query.
Schema information (table names, column names, primary keys, and foreign keys)
is stored separately in \texttt{tables.json} and must be joined with each question
at preprocessing time.

\noindent\textbf{Preprocessing Steps Completed.}
%%>> Aşağıdaki maddeleri gerçek durumunuza göre güncelleyin.
%%>> En azından "dataset downloaded" ve "schema loading" tamamlanmış olmalı.
\begin{itemize}[noitemsep]
  \item Dataset downloaded and directory structure verified.
  \item Schema loading script implemented: \texttt{tables.json} is parsed and
        mapped to each question by \texttt{db\_id}.
  \item Three schema serialization formats (plain text, CREATE TABLE syntax,
        compact notation) have been defined and prototyped.
        %%>> Eğer bunları henüz yazmadıysanız: silin ve "Under development" yazın.
  \item Tokenization pipeline is under development using the HuggingFace
        \texttt{AutoTokenizer} for the selected model.
\end{itemize}

\noindent\textbf{Known Data Challenges.}
The Spider training set exhibits class imbalance across SQL complexity levels;
simple single-table queries are overrepresented relative to nested queries
involving multiple JOINs and subqueries.
Additionally, the test set labels are not publicly released, so all evaluation
is performed on the development split.

We plan to implement two baseline approaches that represent distinct
paradigms for Text-to-SQL generation.

\noindent\textbf{Baseline 1 — Zero-shot Prompting.}
%%>> Uygulama: HuggingFace'ten mistralai/Mistral-7B-Instruct-v0.2 veya
%%>> OpenAI API ile gpt-3.5-turbo kullanın.
%%>> Prompt şablonu:
%%>>   "### Schema: {schema}\n### Question: {question}\n### SQL:"

We will evaluate a large instruction-following language model
(e.g., \texttt{Mistral-7B-Instruct}) in a zero-shot setting.
The model receives the serialized database schema and the natural language
question as a prompt and is asked to generate the SQL query without any
task-specific fine-tuning.
This baseline establishes the upper bound achievable through prompting alone
and serves as a reference for the value added by fine-tuning.

\noindent\textbf{Suitability:} Zero-shot prompting is a standard and widely
reported baseline in recent Text-to-SQL literature, making it a meaningful
point of comparison.

\noindent\textbf{Baseline 2 — Fine-tuned Seq2Seq Model.}
%%>> Uygulama: HuggingFace'ten "tscholak/cxmefzzi" (T5 tabanlı, Spider'da
%%>> fine-tuned, hazır checkpoint) veya kendi Flan-T5-base fine-tune'unuzu

%%>> Bu baseline'ın farkı: fine-tuning VAR ama schema encoding stratejimiz YOK.
The second baseline is a pre-trained sequence-to-sequence model
fine-tuned on Spider without our proposed schema encoding strategy.
We will use a publicly available checkpoint (e.g., \texttt{tscholak/cxmefzzi}
on HuggingFace) to represent the performance of standard fine-tuning,
against which our schema-aware instruction tuning will be compared.

\noindent\textbf{Suitability:} This baseline isolates the contribution of
our schema serialization strategy by holding the model class constant and
varying only the input representation.

\section{Planned Improvements and Technical Direction}

%%>> Bu bölümde ne yapacağınızı net biçimde anlatın.
%%>> Hoca burada "grubun yönü belli mi?" sorusuna cevap arıyor.
%%>> Aşağıdaki paragraflar projenizin genel yönüne uygun — düzenleyebilirsiniz.

Our primary improvement over the baselines is \textbf{schema-aware instruction
tuning}: fine-tuning a sequence-to-sequence language model (Flan-T5-base or
Flan-T5-large) on the Spider training set while providing explicit, structured
schema representations as part of the input.

The planned technical steps are as follows:

\begin{enumerate}[noitemsep]
  \item \textbf{Instruction template design:} Finalize the prompt format that
        includes task instruction, serialized schema, and natural language question.
  \item \textbf{Schema serialization comparison:} Implement and evaluate three
        serialization formats (plain text listing, CREATE TABLE syntax, compact
        notation) to identify the most effective representation.
  \item \textbf{Fine-tuning:} Train Flan-T5-base on Spider using the HuggingFace
        \texttt{Trainer} API with AdamW optimizer, cosine learning rate scheduling,
        and early stopping on development execution accuracy.
  \item \textbf{Inference and evaluation:} Run the fine-tuned model on the Spider
        development set and compute execution accuracy by executing predicted SQL
        against the SQLite databases.
  \item \textbf{Comparison:} Systematically compare the proposed method against
        both baselines and across schema format variants.
\end{enumerate}

If computational resources allow, we will also experiment with LoRA-based
parameter-efficient fine-tuning to reduce GPU memory requirements.

\section{Ablation or Prompt Sensitivity Plan}

%%>> Bu bölümde sadece PLANı anlatın — henüz yapmanıza gerek yok.
%%>> Hoca "ne karşılaştıracaksınız?" sorusunun cevabını arıyor.

Our ablation study will systematically isolate the contribution of the
schema serialization format, which we consider the central design choice
of our system.
Specifically, we will train and evaluate identical model configurations
(same architecture, hyperparameters, and training data) while varying only
the schema representation:

\begin{itemize}[noitemsep]
  \item \textbf{Format A} — plain text listing of tables and columns;
  \item \textbf{Format B} — CREATE TABLE SQL syntax (our primary format);
  \item \textbf{Format C} — compact notation with primary/foreign key annotations.
\end{itemize}

Additionally, if time permits, we plan to compare:
\begin{itemize}[noitemsep]
  \item zero-shot vs.\ few-shot prompting for Baseline 1;
  \item model size effect: Flan-T5-base (250M) vs.\ Flan-T5-large (780M);
  \item with vs.\ without foreign key information in the schema representation.
\end{itemize}

\section{Error Analysis and Generation Quality Analysis}

%%>> Bu bölümde sadece nasıl hata analizi yapacağınızı anlatın.
%%>> Final raporda 15-20 başarısız örnek manuel incelenip
%%>> kategorize edilecek — bunu planı olarak belirtin.

For the final report, we plan to manually inspect a sample of prediction
errors from the Spider development set and categorize them by failure type.
Based on the Text-to-SQL literature, we anticipate the following error categories:

\begin{itemize}[noitemsep]
  \item \textbf{Schema linking errors:} incorrect table or column selection;
  \item \textbf{JOIN errors:} missing or incorrect JOIN conditions and ON clauses;
  \item \textbf{Aggregation errors:} incorrect use of COUNT, SUM, AVG, GROUP BY,
        or HAVING;
  \item \textbf{Nested query failures:} errors in subquery structure;
  \item \textbf{WHERE clause errors:} incorrect filter conditions or string values.
\end{itemize}

For each category, we will provide (1) the count and percentage of failures,
(2) a representative example showing the gold SQL and the predicted SQL,
and (3) a brief explanation of why the model failed.
We also note that evaluating generation quality in Text-to-SQL differs from
open-ended generation: a syntactically fluent query may still be semantically
incorrect, which motivates our use of execution-based evaluation as the
primary metric.

\section{Ethical Considerations and Bias}

%%>> Bu bölüm kısa tutulabilir — 1-2 paragraf yeterli.
%%>> Ama mutlaka yazılmalı (rubrikte 5 puan var).

Deploying Text-to-SQL systems in real-world applications raises several
responsible AI concerns that we acknowledge in our project.

\noindent\textbf{Hallucination risk.}
A model may generate syntactically valid SQL that references non-existent
columns or tables, silently returning incorrect results.
Users without SQL expertise cannot easily detect such errors, making
hallucination particularly risky in this context.

\noindent\textbf{Domain bias.}
The Spider benchmark is English-only and skews toward standard business domains.
Models trained exclusively on Spider may perform poorly on domain-specific
databases (e.g., medical, legal) or on queries formulated in other languages,
raising fairness concerns for non-English-speaking users.

\noindent\textbf{Misuse scenarios.}
A Text-to-SQL system integrated into a database-facing application could
potentially be misused to extract sensitive information if query generation
is not sandboxed or restricted by appropriate access controls.
